# release-publish

## ADDED Requirements

### Requirement: tag-driven image publishing
The repository SHALL run a release workflow (`release.yml`) when a tag matching `v*.*.*` is pushed: it builds the production Docker image and publishes it to `ghcr.io/vezril/dionysus-planner` tagged with the version (`X.Y.Z`, stripped `v`) and `latest`, authenticating only with the workflow's `GITHUB_TOKEN`.

#### Scenario: pushing a version tag publishes the image
- **WHEN** tag `v1.0.0` is pushed
- **THEN** `ghcr.io/vezril/dionysus-planner:1.0.0` and `:latest` become pullable and resolve to the same digest

#### Scenario: no external secrets required
- **WHEN** the release workflow authenticates to GHCR
- **THEN** it uses `GITHUB_TOKEN` with `packages: write` and no repository secret is referenced

### Requirement: version tags are immutable
The release workflow SHALL fail without pushing if the version tag already exists in the registry; `latest` is the only mutable tag.

#### Scenario: re-releasing an existing version is refused
- **WHEN** the workflow runs for `v1.0.0` and `ghcr.io/vezril/dionysus-planner:1.0.0` already exists
- **THEN** the workflow fails before any push and the existing image digest is unchanged

### Requirement: GitHub Release accompanies each version
The release workflow SHALL create a GitHub Release for the tag with generated release notes after the image publish succeeds.

#### Scenario: release created after successful publish
- **WHEN** the image publish for `v1.0.0` succeeds
- **THEN** a GitHub Release named `v1.0.0` exists with autogenerated notes

#### Scenario: failed publish creates no release
- **WHEN** the image build or push fails
- **THEN** no GitHub Release is created for that tag
